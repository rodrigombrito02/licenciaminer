Princípios consolidados na reunião

O que orienta as próximas decisões.

1

Ativos, não reativos

Não esperar cliente bater. Mapear quem precisa, abordar com produto pronto. Site precisa atrair, mas a captação vem de busca ativa.

2

Contexto é tudo

Resultado do Claude depende de quem pergunta saber o que quer. Bons resultados = bons contextos. Treinar o time a pedir bem.

3

IA para nós antes de para fora

Usar a IA pra identificar oportunidades internas (rede, DMs parados, indústrias-alvo) antes de empacotar como produto externo. Isso multiplica capacidade.

4

Sistema (SaaS) ≥ serviço puro

Em muitos casos, oferecer o sistema vale mais que o acompanhamento humano. Aprendizado inicial é parte do produto, não etapa avulsa.

5

Pilotar com poucos antes de escalar

Em vez de produtizar tudo, pegar 1 indústria, 10 DMs, 1 caso de ouro — pacificar processos prioritários antes de abrir frente 2.

6

Cuidar com a preguiça

Lado perigoso: jogamos para a IA o que deveríamos pensar. Equilíbrio: usar pra escalar raciocínio próprio, não para terceirizar pensamento.

7

Rede como produto

Modelo "6 graus de separação" funciona na mineração. Mapear conexões via LinkedIn + universidade + histórico de trabalho é insight de captação.

8

Renomear linha ambiental — "SQ Ambiental" ou "ESP Ambiental"

Decidir até a próxima reunião entre SQ Ambiental e ESP Ambiental. Padronizar em todo o sistema, site e materiais. Termo aberto na reunião.

9

Compartilhamento de contextos entre sócios

Contextos do Claude hoje ficam isolados por usuário; o CloudWorks (chat compartilhado) não usa skills/contextos salvos como Claude Code. Precisa repositório comum pra evitar retrabalho e gasto duplicado de tokens.

10

Freemium nos produtos

Estratégia freemium em Mineral Intelligence e Análise Preliminar: camada gratuita captura o lead via login; versão completa é paga com entregáveis claros.

11

HTML "self-standing" como entrega ao cliente

Apresentações em HTML autocontido (sem link público), arquivos pequenos (~50k), distribuição controlada. Proteção de PI por curadoria de quem recebe.

12

Revisão sistemática das saídas da IA

"Pensa em inglês e traduz" — revisar nomes próprios, termos técnicos, números e traduções PT-BR. Padronizar prompts por cliente (ex.: regras AS Steel — sem "consultês").

Cada sócio mantém uma pasta contextos/[nome]/ no repositório com seus prompts, padrões de pergunta e templates que funcionam. Quando um descobre algo que vale, outros aproveitam.

2

Memória central da Summo

Arquivo MEMORY_SUMMO.md compartilhado com decisões e aprendizados — clientes, padrões, vocabulário "sem consultês", paleta. Cada vez que abrirem o Claude, esse contexto carrega.

3

Templates aplicáveis

Bernardo monta paleta + templates de apresentação. Giulia/Lima/Leo aplicam no Canva ou direto no sistema. Identidade Summo consistente em todos os outputs.

4

Reunião curta semanal de aprendizado

15 min por semana cada um conta uma coisa nova que descobriu da IA — não pra exibir, pra propagar. Bluedot grava, vira insumo do MEMORY_SUMMO.

5

N8N como hub de automação

Alguns já pagam. Compilar ideias e centralizar os fluxos N8N num repositório acessível. Cada automação criada vira ativo compartilhado.

6

Contribuição via GitHub

Cada um clona o repo, propõe melhorias na sua máquina, manda pro Rodrigo via PR. Detalhamento na próxima seção.

Baixar o repositório pra trabalhar localmente. Escolher uma pasta onde gostam de manter projetos.

# abrir terminal (cmd/powershell no Windows ou Terminal no Mac)
cd C:\Projetos      # Windows — ou ~/Projetos no Mac

# clonar (trocar SEU-USUARIO pelo seu username github)
git clone https://github.com/SEU-USUARIO/licenciaminer.git
cd licenciaminer

# adicionar o repositório original como "upstream" (pra puxar atualizações depois)
git remote add upstream https://github.com/rodrigombrito02/licenciaminer.git
3
Instalar dependências (apenas 1ª vez)

O projeto tem duas partes: backend (Python) e frontend (Next.js).

# instalar dependências Python do backend
pip install -e .

# instalar dependências Node do frontend
cd web
npm install
cd ..
4
Rodar o sistema local

Dois terminais abertos ao mesmo tempo — um para o backend, outro para o frontend.

# TERMINAL 1 - backend (FastAPI)
uvicorn api.main:app --reload --port 8000

# TERMINAL 2 - frontend (Next.js)
cd web
npm run dev -- --webpack # --webpack importante! Turbopack está com bug

Abrir no navegador: http://localhost:3000

5
Criar uma branch para suas mudanças

Nunca trabalhar direto no main. Sempre criar uma branch com nome descritivo.

# garantir que está na branch principal e atualizada
git checkout main
git pull upstream main

# criar branch para seu trabalho (exemplo: melhoria no módulo ambiental)
git checkout -b giulia/melhoria-ambiental
# OU: maury/dm-parados, leo/fc-financeiro, lima/produto-financeiro-6

Trabalhar com o Claude Code

Dentro da pasta licenciaminer/, abrir o Claude Code e pedir o que quer mudar. Ele entende a estrutura, edita arquivos, testa.

# exemplo de prompt para o Claude Code

Dica do Maury: contexto bem detalhado = saída boa. Quanto mais específico o pedido, melhor o resultado.

7
Salvar e enviar pro GitHub

Quando achar que está bom, salvar as mudanças e enviar pro seu fork.

# ver o que mudou
git status

# adicionar tudo
git add .

# criar um commit (descrever o que fez)
git commit -m "Adiciona seção de editais do DO-MG na página /ambiental"

# enviar para o GitHub
git push origin giulia/melhoria-ambiental
8
Abrir um Pull Request (PR) para o Rodrigo

O PR é uma solicitação para o Rodrigo revisar e mesclar suas mudanças no sistema oficial.

  1. Após o push, o GitHub mostra um link "Compare & pull request" — clicar nele
  2. Confirmar que vai do seu fork (SEU-USUARIO/licenciaminer:sua-branch) para o repo do Rodrigo (rodrigombrito02/licenciaminer:main)
  3. Escrever o título do PR e uma descrição do que mudou + por que
  4. Clicar em Create pull request
  5. Avisar o Rodrigo no WhatsApp ou no canal do time

Atualizar seu fork periodicamente (a cada 1-2 semanas)

Conforme o Rodrigo incorpora mudanças de todos, vocês precisam puxar essas atualizações.

git checkout main
git pull upstream main
git push origin main      # atualiza o seu fork também

Como o Rodrigo recebe e incorpora

R
Fluxo do Rodrigo ao receber um PR
# 1. Buscar a branch do sócio na minha máquina
git fetch origin pull/[NUMERO-DO-PR]/head:revisao-[nome-socio]

# 2. Trocar pra essa branch e testar
git checkout revisao-[nome-socio]
npm run dev -- --webpack  # testar visualmente
uvicorn api.main:app --reload  # testar backend

# 3. Se aprovado, mesclar via GitHub web (botão "Merge pull request")

# 4. Depois, atualizar minha branch principal local + sprint
git checkout main
git pull origin main
git checkout sprint/melhorias
git merge main

# 5. Push do sprint atualizado
git push origin sprint/melhorias

# 6. Se virou produção, merge sprint → main e push
git checkout main
git merge sprint/melhorias --no-ff
git push origin main
# Vercel detecta e faz deploy automático em ~3-5 minutos
Resumo do fluxo Sócio faz fork → trabalha em branch → abre PR → Rodrigo testa → Rodrigo merge → Vercel publica. Tudo rastreável no histórico do GitHub. Cada PR vira parte da história do sistema.
Erro mais comum (pra evitar) Trabalhar direto no main sem criar branch. Sempre git checkout -b nome/o-que-vou-fazer antes de mexer. Se der ruim, é só apagar a branch e começar de novo.
